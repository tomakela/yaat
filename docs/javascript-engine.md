# Parallel JavaScript engine plan

YAAT's primary runtime target remains the Windows 95-friendly C engine and the deterministic `.yaat` script pipeline. A parallel JavaScript engine is useful as a documentation-backed reference implementation for tooling, browser demos, and rapid iteration, but it must not redefine game behavior independently from the C runtime.

## Goals

- Provide a browser- and Node-compatible engine that can load the same authored game data used by the C runtime.
- Exercise room loading, hotspot dispatch, inventory state, dialogue commands, and room transitions without requiring a Win95 toolchain.
- Serve as an executable specification for script semantics while preserving the C runtime as the compatibility baseline.
- Keep rendering and audio backends replaceable so tests can run headlessly and demos can use Canvas/Web Audio.

## Non-goals

- Do not replace the C engine as the release target for Win95-compatible builds.
- Do not add JavaScript-specific script features that cannot be represented in `.yaat` source or `.yaatbc` bytecode.
- Do not rely on browser timing, random numbers, network access, or host filesystem behavior for script results.
- Do not require the JavaScript engine to load loose development files in exactly the same way as the Win32 runtime; it may use adapters as long as logical asset precedence is preserved.

## Shared data contract

The JavaScript engine should consume the same logical content as the C runtime:

1. Loose development assets from `game/` during local authoring.
2. Patch archives in descending `patchNNNN.dat` order.
3. The base `game.dat` archive.
4. `.yaatbc` bytecode packages generated by the shared compiler path.

When direct archive access is unavailable, a host adapter may provide an already-enumerated manifest and byte buffers. The adapter must still expose the same winning asset for a given path as the documented runtime precedence rules.

## Architecture

Suggested modules:

- `assetStore`: resolves paths through loose-file, patch, and base-archive adapters.
- `bytecodeLoader`: reads `.yaatbc` packages using explicit little-endian decoding and fixed-width string handling.
- `scriptVm`: executes the same command opcodes as the C VM and returns structured effects.
- `gameState`: owns current room, inventory, flags, visible object state, and save/load serialization.
- `renderer`: abstracts Canvas, test snapshots, or future native wrappers.
- `audio`: abstracts Web Audio, muted test playback, or future native wrappers.
- `input`: translates clicks, pointer movement, keyboard shortcuts, and inventory selections into engine events.

Keep these modules independent from DOM globals. Browser demos should compose adapters at the edge rather than importing `window` or `document` from core runtime modules.

## Determinism rules

The JavaScript engine must match the deterministic behavior expected from the C runtime:

- Execute command lists strictly in bytecode order.
- Treat all script-visible state as explicit `gameState` data.
- Return requested side effects, such as dialogue, sounds, object visibility changes, and room transitions, instead of performing hidden mutations in rendering or audio backends.
- Avoid floating-point values in script-visible state unless a future bytecode version defines exact serialization and comparison rules.
- Gate timers, animation clocks, and audio callbacks outside the script VM so they cannot alter script branching directly.

## Testing strategy

Use shared fixtures to compare JavaScript behavior with the C implementation:

- Load the same `.yaat` fixtures after compilation to `.yaatbc`.
- Snapshot decoded rooms, entities, events, and command records.
- Run scripted event traces and compare resulting state, emitted effects, and diagnostics.
- Include archive precedence tests that mirror the runtime asset-store checks.
- Keep browser-only rendering tests separate from VM conformance tests.

## Documentation responsibilities

Any future JavaScript implementation should update the relevant docs when behavior changes:

- Update `docs/script-bytecode.md` for package format changes.
- Update `docs/script-runtime.md` for VM semantics shared by C and JavaScript.
- Update `docs/asset-structure.md` for asset precedence or adapter expectations.
- Update this document for JavaScript-only host integration details.
